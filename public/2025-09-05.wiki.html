How do I use Vim? (Vim Carnival September 2025)

I still use Old Vim — it’s not just a text editor; it’s a philosophy wrapped in keystrokes. There’s no onboarding, no curated experience. Just you, a blinking cursor, and the manpages if you dare. It’s stubborn, yes—but gloriously so.

Lua? I pass — I prefer the cryptic charm of Vimscript. I’ve looked at the plugin-rich world of Neovim with a kind of puzzled awe—like standing before a lavish buffet I never ordered from. It’s overflowing with features, integrations, and abstractions that feel distant from the simplicity I value. Vim, in contrast, offers a raw, direct experience that stays true to its original philosophy—lean, consistent, and enduring.

Minimalism in tooling it’s about clarity. Every tool you add is a trade-off: more power, yes, but also more friction, and more updates. I prefer tools that do one thing well and disappear when not needed. The fewer moving parts, the fewer things that break. And when something does go wrong, you actually understand why. That’s the beauty of minimalism.

My setup

I keep a single .vimrc—almost readable, half-commented, and lovingly chaotic.

I run Vim from Windows PowerShell, using zoxide for fuzzy directory jumps. A couple of small Vim9script plugins handle quick tasks by talking to PowerShell directly. For file and search navigation, I use fd and ripgrep.

Clipboard integration is handled via the + register, with Ditto providing clipboard history. An AutoHotKey script remaps Caps Lock to Esc — no pinky yoga required, just blissful normal‑mode comfort.

My dotfiles travel light, synced through a bare Git repo. No Lua. No Node.

Language Server, My Way

I set up LSP manually using Yegappan’s Vim9script plugin. I register each server explicitly in my .vimrc, specifying the filetypes, executable path, and arguments. It’s declarative, readable, and doesn’t hide anything behind abstractions.

Git and Coreutils: Native, Minimal, No WSL

I use the precompiled GNU tools bundled with the mingw-w64 distribution—no need to compile anything myself. Git and coreutils come ready to go, and I simply add their paths to my PowerShell environment. This gives me access to familiar commands like ls, awk, and less without installing WSL, Cygwin, or any heavyweight Unix layer.

This setup is clean and self-contained. Just a few binaries and a $env:Path tweak. It’s Unix-like enough for scripting, but native to Windows—exactly the balance I want. And because it includes the Git command line, I can use plugins like fugitive and gitgutter without friction. No wrappers, no emulation—just direct, reliable integration with the tools I already trust.

Overcome Limitations

Using Old Vim is like riding a well‑oiled bicycle — not a flashy e‑bike with a touchscreen, but reliable, fast, and easy to fix with a wrench. Its limits push you to learn clever, composable tricks instead of outsourcing everything to a single plugin.

Think of Vim9script as swapping a better set of gears onto that bike. Vim9script is, in my humble opinion, the sensible upgrade Vim deserved: faster, clearer, and actually readable. Bram didn’t just start and maintain Vim — he refined it, and Vim9script feels like his greatest gift.
